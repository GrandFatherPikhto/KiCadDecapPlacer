# Модули верхнего уровня

## 1. `kicadspoke_cli.py` — точка входа (CLI)

### Назначение

Главный исполняемый скрипт, который обрабатывает аргументы командной строки, инициализирует логирование, загружает конфигурацию, подключается к KiCad, выполняет предварительную валидацию, запускает планировщик и исполнитель, а также поддерживает команду `undo` для отката последней операции.

### Основные функции

| Функция | Описание |
|---------|----------|
| `setup_logging(verbose, log_file)` | Настраивает логирование: уровень (INFO/DEBUG), вывод в консоль и/или файл. |
| `cmd_apply(args)` | Основная команда `apply`: загружает конфиг, подключается к KiCad, запускает валидацию, планирование и двухфазное исполнение (перемещения → refresh → via). Обрабатывает `--dry-run`. |
| `cmd_undo(args)` | Команда `undo`: находит последний JSON-лог в папке `logs/`, вызывает `undo_last_operation()`. |
| `main()` | Парсит аргументы (поддерживает неявный `apply`), настраивает логирование, вызывает соответствующую команду, перехватывает исключения. |

### Ключевые зависимости

- `config.load_config` – загрузка конфига.
- `kicad.adapter.KiCadBoardAdapter` – подключение к KiCad.
- `validation.run_all_checks` – предварительная проверка.
- `placement.planner.PlacementPlanner` – планирование.
- `placement.executor.BatchExecutor` – исполнение.
- `undo.undo_last_operation` – откат.

### Особенности

- **Двухфазное выполнение**: после `execute_moves()` вызывается `adapter.refresh_board()`, затем `plan_vias()` и `execute_vias()`. Это гарантирует, что via (особенно термовиа) рассчитываются по реальному состоянию платы.
- **Dry-run**: в этом режиме via не показываются, так как их позиции зависят от реального размещения компонентов.
- **Обработка ошибок**: все исключения перехватываются и логируются; пользовательские (`PlacerError`) выводятся без стека.

## 2. `config.py` — загрузка и хранение конфигурации

### Назначение

Определяет все структуры данных (dataclass'ы), описывающие конфигурацию расстановки, и предоставляет функцию `load_config()` для загрузки YAML-файла в типизированные объекты Python. Включает проверку уникальности ролей внутри шаблона (фатальная ошибка при дублировании).

### Основные классы (dataclass'ы)

| Класс | Описание |
|-------|----------|
| `ThermalViaArrayConfig` | Настройки массива термовиа под термопадом. |
| `TemplateVia` | Описание via в шаблоне (локальные координаты, цепь, размеры). |
| `TemplateComponentSlot` | Слот компонента в шаблоне: роль, локальные координаты, угол, список via. |
| `SpokeTemplate` | Полный шаблон спицы: имя, список via (уровня спицы), список слотов компонентов. |
| `ManualSpoke` | Конкретная спица: пад, шаблон, сдвиг, поворот, флаг `enabled`. |
| `Rule` | Правило для одной цепи: имя цепи и список спиц. |
| `Config` | Главный объект: целевой компонент, сторона, шаблоны, термовиа, правила, флаги. |

### Основные функции

| Функция | Описание |
|---------|----------|
| `load_config(path)` | Читает YAML, парсит все секции, возвращает объект `Config`. При загрузке шаблонов проверяет уникальность ролей в `components`; при дублировании бросает `ValidationError`. |
| `_load_template_via(data)` | Загружает `TemplateVia` из словаря. |
| `_load_template_component_slot(data)` | Загружает `TemplateComponentSlot`. |
| `_load_spoke_template(name, data)` | Загружает `SpokeTemplate` с проверкой уникальности ролей. |
| `_load_manual_spoke(data)` | Загружает `ManualSpoke`. |

### Особенности

- **Проверка уникальности ролей**: внутри одного шаблона роли должны быть уникальны (иначе невозможно однозначно сопоставить роль и refdes). Ошибка выводится через `format_fatal_error()`.
- **Наследование параметров via**: если `net` не указан в `TemplateVia`, используется цепь из `rule.net`.

## 3. `exceptions.py` — иерархия исключений

### Назначение

Определяет пользовательские исключения для проекта и единую функцию форматирования фатальных ошибок. Все исключения наследуются от базового `PlacerError`.

### Классы исключений

| Класс | Назначение |
|-------|------------|
| `PlacerError` | Базовое исключение для всех ошибок планера. |
| `BoardNotFoundError` | Не удалось получить плату из KiCad. |
| `ComponentNotFoundError` | Компонент не найден на плате. |
| `GeometryError` | Ошибка в геометрических расчётах. |
| `ValidationError` | Фатальная ошибка предварительной проверки конфигурации — программа останавливается до изменения платы. |

### Вспомогательная функция

| Функция | Описание |
|---------|----------|
| `format_fatal_error(title, problems)` | Форматирует список проблем в единое многострочное сообщение с рамкой из `=`. Используется как в `config.py` (проверки на этапе загрузки YAML), так и в `validation.py` (проверки после подключения к KiCad). Живёт здесь, чтобы избежать циклических импортов. |

## 4. `undo.py` — откат последней операции

### Назначение

Реализует команду `undo`, которая восстанавливает состояние платы до выполнения последней операции расстановки. Использует JSON-логи, создаваемые `executor.py` при каждом успешном применении изменений.

### Основная функция

| Функция | Описание |
|---------|----------|
| `undo_last_operation(json_path)` | Загружает JSON-лог, для каждого перемещённого компонента: определяет исходный слой (по строке), при необходимости выполняет флип, затем восстанавливает позицию и угол. Для каждой созданной via удаляет её по UUID. После успешного отката удаляет JSON-файл. |

### Алгоритм восстановления слоя

- `original_layer` хранится в логе как строка `"F.Cu"` или `"B.Cu"`.
- Если текущий слой футпринта отличается от исходного, выполняется `adapter.flip_selected([fp])`, затем футпринт перечитывается через `adapter.get_footprint(ref)` для обновления объекта.
- Затем восстанавливаются позиция и угол.

### Используемые API

- `adapter.flip_selected()` – переворот компонента.
- `adapter.update_items()` – обновление позиции и угла.
- `board.remove_items_by_id()` – удаление via по UUID (использует `kipy.proto.common.types.KIID`).

## 5. `validation.py` — предварительные проверки конфигурации

### Назначение

Выполняет фатальные проверки конфигурации **до** любых изменений на плате. Если обнаружена проблема — программа останавливается с подробным списком ошибок, не трогая плату.

### Основные функции

| Функция | Описание |
|---------|----------|
| `check_templates_and_pads_exist(adapter, cfg)` | Проверяет, что каждая включённая спица ссылается на существующий шаблон и существующую площадку целевого компонента. Пропускает отключённые спицы (`enabled=False`). |
| `check_role_pool_sufficiency(adapter, cfg)` | Для каждого правила строит `ComponentPool` и сверяет требуемое количество компонентов каждой роли с доступным. Если не хватает — сообщает все нехватки разом. |
| `run_all_checks(adapter, cfg)` | Запускает обе проверки по порядку. Вызывается из `cmd_apply()` после подключения к KiCad. |

### Особенности

- **Сбор всех проблем**: проверки собирают список ошибок, а не останавливаются на первой. Это позволяет пользователю увидеть все проблемы конфигурации за один запуск.
- **Использование `ComponentPool`**: в `check_role_pool_sufficiency` строится пул для каждой цепи, чтобы проверить наличие компонентов по ролям. Это дублирует логику `ManualPositionCalculator`, но выполняется до любых изменений.
- **Форматирование**: ошибки выводятся через `format_fatal_error()` из `exceptions.py`.

---

## Взаимосвязи

- `kicadspoke_cli.py` импортирует `load_config` из `config.py`, `undo_last_operation` из `undo.py`, `run_all_checks` из `validation.py`, а также классы адаптера, планировщика и исполнителя.
- `config.py` использует `ValidationError` и `format_fatal_error` из `exceptions.py` для проверки уникальности ролей.
- `validation.py` также использует `format_fatal_error` и `ComponentPool` из `placement/services/component_pool.py`.
- `undo.py` использует `KiCadBoardAdapter` и `BoardLayer`.

Все эти модули составляют каркас программы: CLI, конфигурация, обработка ошибок, валидация и откат. Они не содержат бизнес-логики расстановки (которая находится в `placement/`), но обеспечивают её корректный запуск и завершение.
