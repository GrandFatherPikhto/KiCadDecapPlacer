# Общая архитектура проекта KiCadSpoke

## 1. Назначение

KiCadSpoke — это инструмент командной строки для **автоматизированной расстановки компонентов и переходных отверстий** в печатных платах, разработанных в среде **KiCad**. Программа подключается к открытому экземпляру KiCad через IPC (Inter-Process Communication) и выполняет:

- Перемещение компонентов (конденсаторов, резисторов, кварцев и т.д.) на заданные позиции согласно шаблонам «спиц».
- Создание переходных отверстий (via) на основе геометрии шаблонов.
- Откат последней операции (undo) с восстановлением исходного состояния платы.

В основе подхода лежит понятие **спицы** (spoke) — набора компонентов и via, физически относящихся к одному выводу микросхемы. Вся геометрия спицы описывается в **локальных координатах** в виде **шаблона**, который можно применить к любому паду микросхемы, задав сдвиг и поворот.

---

## 2. Принципы архитектуры

Архитектура построена на следующих принципах:

| Принцип | Описание |
|---------|----------|
| **Разделение ответственности** | Чёткое разделение на слои: CLI, бизнес-логика, доступ к KiCad, геометрические утилиты, конфигурация. |
| **Инкапсуляция сложности IPC** | Все вызовы к KiCad сосредоточены в адаптере, который предоставляет простой интерфейс. |
| **Расширяемость** | Новые типы via, компонентов или шаблонов легко добавляются за счёт модульной структуры. |
| **Безопасность** | Двухфазное выполнение (сначала перемещения, потом via) с промежуточным перечитыванием платы; валидация перед любыми изменениями; логирование и откат. |
| **Идемпотентность** | Повторный запуск не создаёт дублирующих via и не перемещает уже стоящие на месте компоненты. |
| **Детерминизм** | Автоматический подбор компонентов из пула по ролям, сортировка по естественному числовому порядку. |

## 4. Слои архитектуры (снизу вверх)

### 4.1. Уровень доступа к KiCad (`kicad/`)

- **`adapter.py`** – реализует `KiCadBoardAdapter`, инкапсулирующий все вызовы к библиотеке `kipy`. Предоставляет методы для поиска компонентов, падов, зон, цепей, получения bounding box'ов, транзакций (commit/rollback), создания via и выполнения GUI-действий (flip). Устойчив к ошибкам «KiCad is busy» (повторные попытки).
- **`interfaces.py`** – определяет абстрактный интерфейс `IBoardAdapter`, позволяющий легко подменять адаптер (например, для тестирования с моками).
- **`__init__.py`** – экспортирует публичные классы.

### 4.2. Геометрические утилиты (`geometry/`)

Модули, работающие с координатами, не зависящие от KiCad:

- **`keepout.py`** – класс `Rect` (AABB-прямоугольник) и функции для построения keepout-областей, проверки точек на занятость, поиска свободного места.
- **`pad_projection.py`** – предсказание позиции пада после перемещения и поворота компонента с учётом флипа (зеркалирования).
- **`spoke_layout.py`** – преобразование локальных координат шаблона спицы (`along/across`) в абсолютные координаты платы с учётом сдвига и поворота. Также генерирует via (как на уровне спицы, так и на уровне компонента).
- **`thermal_grid.py`** – генерация сетки термовиа под площадкой.
- **`placement.py`** – устаревший файл, оставлен для обратной совместимости.

### 4.3. Бизнес-логика (`placement/`)

Содержит основную логику планирования и исполнения:

- **`planner.py`** – класс `PlacementPlanner`, главный координатор. Вызывает `ManualPositionCalculator` для расчёта позиций компонентов и via, затем формирует команды для перемещения и создания via. Также содержит логику пропуска уже стоящих на месте компонентов.
- **`executor.py`** – класс `BatchExecutor`, применяет команды к плате через адаптер. Разделён на две фазы: `execute_moves()` и `execute_vias()`. Ведёт логирование операций для undo.
- **`commands.py`** – структуры данных (`MoveCommand`, `ViaCommand`, `PlacedComponentInfo`) для передачи между модулями.
- **`collision.py`** – упрощённая проверка коллизий между компонентами (по кругам-приближениям).
- **`services/`**:
  - **`component_pool.py`** – класс `ComponentPool`, сбор компонентов по ролям (`Role`) и цепи, сортировка, раздача спицам.
  - **`manual_position_calculator.py`** – основной сервис, который для каждого правила строит пул компонентов, затем для каждой спицы применяет `apply_spoke_geometry` и возвращает готовые `PlacedComponentInfo` и `ViaCommand` (все via рассчитываются заранее, без обращения к живой плате).
  - **`via_planner.py`** – отвечает только за термовиа и фильтрацию существующих via при `skip_existing_components`. Все остальные via уже рассчитаны в `ManualPositionCalculator`.

### 4.4. Конфигурация и валидация

- **`config.py`** – определяет датаклассы для всех структур конфига (`Config`, `SpokeTemplate`, `ManualSpoke`, `TemplateVia`, `TemplateComponentSlot` и т.д.) и функцию `load_config()` для загрузки YAML.
- **`validation.py`** – предварительные проверки: существование шаблонов и падов, достаточность компонентов в пуле по ролям. Бросает `ValidationError` с понятным списком проблем.

### 4.5. Утилиты и исключения

- **`utils/units.py`** – константа `MM = 1_000_000` для перевода миллиметров в нанометры.
- **`exceptions.py`** – иерархия пользовательских исключений, включая `ValidationError` и функцию форматирования `format_fatal_error`.

### 4.6. Откат операций (undo)

- **`undo.py`** – функция `undo_last_operation()`, которая читает последний JSON-лог и восстанавливает плату (возвращает компоненты на исходные позиции и слой, удаляет созданные via).

### 4.7. Пользовательский интерфейс (CLI)

- **`kicadspoke_cli.py`** – главный исполняемый файл, обрабатывающий аргументы командной строки, вызывающий загрузку конфига, подключение к KiCad, валидацию, планирование и выполнение, а также команду undo.

### 4.8. Диагностика (`diagnostics/`)

- Скрипты для отладки: `get_pad_bbox.py`, `get_power_via.py`, `test_custom_field.py`. Могут быть полезны для проверки геометрии и чтения полей.

## 5. Взаимодействие модулей (последовательность работы)

```
1. Запуск CLI (kicadspoke_cli.py)
   │
   ├── Разбор аргументов (argparse)
   ├── Настройка логирования
   │
2. Загрузка конфига (load_config)
   │
   ├── Чтение YAML
   ├── Проверка уникальности ролей в шаблонах
   ├── Создание объекта Config
   │
3. Подключение к KiCad
   │
   ├── Создание KiCadBoardAdapter
   ├── adapter.refresh_board() — получение платы
   │
4. Валидация (run_all_checks)
   │
   ├── check_templates_and_pads_exist()
   ├── check_role_pool_sufficiency()
   │   └── Строит ComponentPool для каждого правила,
   │       проверяет количество компонентов по ролям
   │
5. Планирование (PlacementPlanner)
   │
   ├── plan_moves()
   │   ├── Для каждого правила строит ComponentPool
   │   ├── Для каждой спицы:
   │   │   ├── pop() — берёт refdes для каждой роли
   │   │   ├── apply_spoke_geometry() — вычисляет позиции
   │   │   │   └── генерирует ViaCommand (via уровня спицы и компонента)
   │   │   └── формирует PlacedComponentInfo
   │   └── Возвращает списки MoveCommand и ViaCommand
   │
   ├── (Если --dry-run — вывод плана и выход)
   │
6. Исполнение (двухфазное)
   │
   ├── Фаза 1: перемещения
   │   ├── executor.execute_moves()
   │   │   ├── Сбор исходных состояний (для undo)
   │   │   ├── Проверка коллизий (опционально)
   │   │   ├── Флип (если слой изменился)
   │   │   ├── Перемещение батчами через commit_with_retry
   │   │   └── Сохранение лога перемещений
   │   │
   │   ├── adapter.refresh_board() — перечитываем плату
   │   │
   ├── Фаза 2: via
   │   ├── planner.plan_vias()
   │   │   ├── Фильтрация существующих via (skip_existing_components)
   │   │   ├── Генерация термовиа (с поиском свободных мест)
   │   │   └── Возвращает итоговый список ViaCommand
   │   │
   │   ├── executor.execute_vias()
   │   │   ├── Создание via батчами через commit_with_retry
   │   │   ├── Сбор UUID созданных via (для undo)
   │   │   └── Запись единого JSON-лога операции
   │   │
   └── Вывод результата (успех/ошибки)
```

## 6. Ключевые архитектурные решения

| Решение | Обоснование |
|---------|-------------|
| **Отказ от зависимости via от живой платы** | Все via (кроме термовиа) вычисляются исключительно по геометрии шаблона. Это делает их независимыми от текущего состояния платы и позволяет планировать их до перемещений. |
| **Двухфазное выполнение** | Необходимо для термовиа (чтобы не мешать перемещениям) и для корректности `skip_existing_components` (via создаются после перемещений, чтобы проверить их наличие в финальных позициях). |
| **Идемпотентность** | Компоненты и via пропускаются, если уже находятся в целевых позициях (с допуском 0.01 мм). Это позволяет безопасно перезапускать скрипт. |
| **Валидация на раннем этапе** | Предотвращает случайное изменение платы при ошибочном конфиге. |
| **Автоматический подбор refdes по ролям** | Устраняет необходимость вручную прописывать имена компонентов в конфиге, делая его компактным и устойчивым к изменениям нумерации. |
| **Логирование и undo** | Каждая операция сохраняется в JSON, что позволяет откатить изменения в случае ошибки. |
| **Модульные тесты** | Все ключевые сценарии покрыты тестами на моках (без реального KiCad), обеспечивая регрессионную защиту. |

## 7. Зависимости и требования

- **Python 3.9+**
- **KiCad 8.0+** с включённым IPC-сервером.
- **Библиотека `kipy`** – официальный Python-клиент для IPC KiCad.
- Дополнительные пакеты: `pyyaml`, `sexpdata`, `numpy`, `scipy` (для тестов и диагностики).

---

## 8. Заключение

Архитектура KiCadSpoke обеспечивает:

- **Гибкость** – легко добавлять новые шаблоны, роли и типы via.
- **Надёжность** – двухфазное выполнение, валидация, идемпотентность и undo.
- **Удобство использования** – конфигурация без явных refdes, автоматический подбор компонентов.
- **Тестируемость** – чёткое разделение слоёв и модульные тесты.

Это делает инструмент пригодным для использования в реальных проектах по разработке печатных плат.
